# 🤖 AI로 예상 질문 생성하기 - 프롬프트 템플릿

> **목표**: ChatGPT, Claude 등 AI를 활용하여 발주처 예상 질문 및 답변을 준비하기

---

## 🎯 왜 예상 질문 준비가 중요한가?

### 회의/PT에서 질문 대응의 중요성

```
✅ 신뢰 구축: 명확한 답변 → 전문성 인정
✅ 차별화: 경쟁사 대비 준비된 모습
✅ 리스크 감소: 돌발 질문 대응 능력
✅ 자신감: 연습을 통한 실전 대응력 향상
```

### AI 활용의 장점

```
1. 다양한 관점: 공무원 입장에서 질문 생성
2. 빠른 속도: 30개 질문을 5분 내 생성
3. 카테고리별 분류: 기술, 예산, 일정 등
4. 답변 힌트 제공: 모범 답변 구조 제시
```

## 📝 예상 질문 생성 프롬프트 템플릿

### 프롬프트 1: 기본 예상 질문 30개 생성

```
당신은 공공기관 IT 프로젝트 평가위원입니다.
20년 경력의 공무원이며, 발주처 입장에서 질문합니다.

다음 제안서 정보를 바탕으로
PT 또는 회의에서 물어볼 가능성이 높은 질문 30개를 생성해주세요.

[제안서 정보]
- 발주기관: [예: 서울시 교육청]
- 과제명: [예: 학생 관리 시스템 구축]
- 예산: [예: 1억원]
- 기간: [예: 6개월]
- 회사 규모: [예: 중소기업, 직원 10명]
- 기술 스택: [예: Java, Spring Boot, Vue.js, MySQL]
- 유사 실적: [예: 민간 프로젝트 2건]

[질문 카테고리 및 개수]
1. 사업 이해도 (5개)
2. 기술 역량 (10개)
3. 프로젝트 수행 능력 (5개)
4. 일정 및 인력 (5개)
5. 비용 및 유지보수 (5개)

[질문 난이도]
- 일반 질문: 50% (확인 차원)
- 검증 질문: 30% (전문성 확인)
- 공격형 질문: 20% (약점 지적)

[질문자 특성]
- 리스크 회피 성향 (실패 두려워함)
- 예산 초과 우려
- 보안 사고 걱정
- 비개발자 (기술 용어 이해 어려움)

[출력 형식]
각 질문에 대해 다음 정보 포함:
1. 질문 내용
2. 질문 의도 (왜 이 질문을 하는가?)
3. 답변 핵심 포인트 (3가지)
4. 예시 답변 (간략히)
```

**예시 출력**
```markdown
## 예상 질문 30개

### 1. 사업 이해도 (5개)

**Q1. 본 사업의 가장 중요한 목표가 무엇이라고 생각하시나요?**

질문 의도:
- RFP를 제대로 읽었는지 확인
- 발주기관의 진짜 니즈를 이해하는지 검증

답변 핵심 포인트:
1. RFP의 사업 목적 정확히 인용
2. 발주기관의 현재 문제점 공감
3. 해결 방안 간략히 제시

예시 답변:
"RFP에 명시된 대로, 본 사업의 핵심 목표는
'학교별로 분산된 학생 정보를 통합 관리하여 업무 효율성을 높이는 것'입니다.
현재 각 학교가 독립적으로 관리하여 발생하는 데이터 중복, 수작업 과다 등의
문제를 해결하는 것이 핵심입니다..."

---

**Q2. 우리 기관의 특성을 얼마나 이해하고 계신가요?**

질문 의도:
- 발주기관에 대한 사전 조사 여부 확인
- 맞춤형 제안인지, 복붙인지 판단

답변 핵심 포인트:
1. 발주기관 규모, 특성 언급 (사전 조사한 내용)
2. 타 기관과의 차이점 강조
3. 맞춤형 제안임을 강조

예시 답변:
"서울시 교육청은 약 1,200개 초중고를 관할하는 대규모 조직으로,
타 교육청 대비 학생 수가 2배 많고, 시스템 복잡도도 높습니다.
따라서 단순한 CMS가 아닌, 대용량 처리와 학교별 권한 관리가
가능한 시스템이 필요하다고 판단했습니다..."
```

### 프롬프트 2: 약점 공격형 질문 생성

```
당신은 공공기관 IT 프로젝트 평가위원이며,
제안사의 약점을 날카롭게 지적하는 역할입니다.

다음 제안사의 약점을 바탕으로
날카로운 질문 10개를 생성해주세요.

[제안사 약점]
- 회사 설립 2년차 (역사 짧음)
- 공공 프로젝트 실적 없음 (민간만 2건)
- 직원 10명 (소규모)
- 백엔드 경험 적음 (프론트 위주)
- 대표가 비개발자 출신

[질문 유형]
- 경험 부족 지적
- 회사 규모 의심
- 기술 역량 검증
- 프로젝트 수행 능력 의심
- 유지보수 우려

[질문 톤]
- 직접적이고 날카로운 질문
- "정말 가능한가?" 식의 의심 표현
- 압박 질문

[출력 형식]
각 질문에 대해:
1. 질문 내용 (공격적)
2. 이 질문이 왜 어려운지
3. 대응 전략 (어떻게 답변할 것인가)
4. 모범 답변 예시
```

**예시 출력**
```markdown
## 약점 공격형 질문 10개

**Q1. 회사 설립한 지 2년밖에 안 됐는데, 이런 큰 프로젝트를 감당할 수 있나요?**

왜 어려운 질문인가:
- 회사 역사가 짧다는 약점 직접 공격
- 신뢰도 의심
- 프로젝트 완수 능력 의심

대응 전략:
1. 회사는 짧아도, 인력 경력은 충분함 강조
2. 소규모이기에 오히려 집중도 높음 재포장
3. 협력사 네트워크로 리스크 보완
4. 구체적인 안전장치 제시

모범 답변:
"회사 설립은 2년이지만, 핵심 인력의 평균 경력은 5년 이상입니다.
대표와 CTO는 대기업 출신으로 공공 프로젝트 경험이 풍부합니다.

오히려 작은 규모이기에 본 프로젝트에 전사적으로 집중 가능하며,
대기업처럼 동시에 20개 프로젝트를 진행하지 않습니다.

또한 만약의 상황에 대비해:
1. 소스코드 에스크로 (제3자 보관)
2. 협력사와 MOU 체결 (대체 수행 가능)
3. 손해배상보험 가입

이러한 안전장치로 리스크를 최소화했습니다."

---

**Q2. 공공 프로젝트 경험이 없는데, 괜찮을까요?**

왜 어려운 질문인가:
- 핵심 약점 (실적 없음) 정면 공격
- 민간 vs 공공의 차이 이해도 의심
- 실패 가능성 우려

대응 전략:
1. 민간 프로젝트 경험의 유사성 강조
2. 공공 프로젝트 전문가와 협력 명시
3. 준비도 (학습, 조사) 강조
4. 오히려 첫 프로젝트라 더 집중할 것임

모범 답변:
"공공 프로젝트는 처음이지만, 유사한 민간 프로젝트 경험이 있습니다.

[민간 프로젝트 실적]
- A기업 학생 관리 시스템 (본 과제와 90% 유사)
- B기업 CMS (게시판, 권한 관리 등 동일 기능)

[공공 프로젝트 준비]
- 전자정부 표준 프레임워크 학습 완료
- 행안부 보안 가이드 숙지
- 공공 프로젝트 경험자 (20건 이상)를 기술 자문으로 영입

오히려 첫 공공 프로젝트이기에
레퍼런스를 만들기 위해 더욱 신중히 수행하겠습니다."
```

### 프롬프트 3: 기술 검증 질문 생성

```
당신은 공공기관 IT 프로젝트 평가위원이며,
정보처리기사 자격증을 보유한 기술 전문가입니다.

다음 기술 스택에 대한 깊이 있는 기술 질문 15개를 생성해주세요.

[기술 스택]
- 백엔드: Java 17, Spring Boot 3.2, Spring Security
- 프론트엔드: Vue.js 3, TypeScript
- 데이터베이스: MySQL 8.0
- 배포: Docker, Docker Compose
- 보안: HTTPS, ARIA 256 암호화

[질문 카테고리]
1. 아키텍처 설계 (3개)
2. 보안 (4개)
3. 성능 및 확장성 (3개)
4. 데이터베이스 (3개)
5. 배포 및 운영 (2개)

[질문 난이도]
- 기본: 40% (개념 이해 확인)
- 중급: 40% (실무 경험 확인)
- 고급: 20% (깊이 있는 이해 확인)

[출력 형식]
각 질문에 대해:
1. 질문 내용
2. 질문 난이도 (기본/중급/고급)
3. 올바른 답변 (전문적이고 구체적으로)
4. 잘못된 답변 예시 (이렇게 답하면 안 됨)
```

**예시 출력**
```markdown
## 기술 검증 질문 15개

### 1. 아키텍처 설계 (3개)

**Q1. 3-Tier 아키텍처를 적용한다고 했는데, 각 계층의 역할을 설명해주세요.**

난이도: 기본

올바른 답변:
"3-Tier 아키텍처는 다음 3개 계층으로 구성됩니다:

1. Presentation Layer (프론트엔드)
   - 역할: 사용자 인터페이스, 입력 검증
   - 기술: Vue.js, HTML/CSS

2. Business Logic Layer (백엔드)
   - 역할: 비즈니스 로직 처리, API 제공
   - 기술: Spring Boot, Java
   - 구조: Controller → Service → Repository

3. Data Layer (데이터베이스)
   - 역할: 데이터 저장 및 관리
   - 기술: MySQL

각 계층이 독립적으로 개발/배포 가능하여
유지보수성과 확장성이 높습니다."

잘못된 답변:
"3-Tier는 프론트, 백엔드, DB를 분리하는 겁니다"
→ 너무 간략, 구체성 부족

---

**Q2. RESTful API 설계 시 CRUD를 어떤 HTTP 메서드로 매핑하나요?**

난이도: 기본

올바른 답변:
"RESTful API는 다음과 같이 HTTP 메서드를 사용합니다:

- CREATE (생성): POST /api/users
- READ (조회): GET /api/users (전체), GET /api/users/{id} (단건)
- UPDATE (수정): PUT /api/users/{id} (전체 수정) 또는 PATCH (일부 수정)
- DELETE (삭제): DELETE /api/users/{id}

또한 적절한 HTTP 상태 코드를 반환합니다:
- 200 OK: 조회/수정 성공
- 201 Created: 생성 성공
- 204 No Content: 삭제 성공
- 400 Bad Request: 잘못된 요청
- 404 Not Found: 리소스 없음
- 500 Internal Server Error: 서버 오류"

잘못된 답변:
"모두 POST로 처리합니다"
→ RESTful 원칙 위반
```

### 프롬프트 4: 시나리오 기반 질문 생성

```
당신은 공공기관 IT 프로젝트 평가위원입니다.

다음 시나리오 상황에서 제안사에게 물어볼 질문 10개를 생성해주세요.

[시나리오]
프로젝트 진행 중 다음과 같은 문제가 발생했습니다:
1. 개발 도중 요구사항이 30% 변경됨
2. 핵심 개발자 1명이 퇴사함
3. 서버 장애로 시스템 2시간 중단됨
4. 보안 취약점이 발견됨
5. 일정이 1개월 지연될 것 같음

[질문 유형]
- "이런 상황이 발생하면 어떻게 대응하나요?"
- "예방 대책은 무엇인가요?"
- "과거 유사 경험이 있나요?"

[출력 형식]
각 시나리오별로:
1. 질문 2개
2. 답변 전략 (어떻게 답할 것인가)
3. 모범 답변 (구체적으로)
```

**예시 출력**
```markdown
## 시나리오 기반 질문 10개

### 시나리오 1: 요구사항 30% 변경

**Q1. 개발 중에 요구사항이 크게 변경되면 어떻게 대응하시나요?**

답변 전략:
- 유연한 개발 방법론 강조 (Agile)
- 단계별 검수로 조기 발견 가능
- 변경 영향도 분석 → 일정/비용 조정 협의

모범 답변:
"요구사항 변경은 프로젝트에서 흔히 발생합니다.
저희는 다음과 같이 대응합니다:

1. 변경 영향도 분석 (1-2일 소요)
   - 어떤 기능에 영향 있는지 파악
   - 추가 개발 기간 산정

2. 발주처와 협의
   - 일정 연장 또는 기능 우선순위 조정
   - 추가 비용 발생 시 투명하게 공유

3. 유연한 개발 방식
   - Agile 방법론으로 2주 단위 스프린트
   - 각 스프린트마다 발주처 검수 → 조기 피드백

실제로 XX프로젝트에서도 요구사항이 20% 변경되었으나,
단계별 검수 덕분에 조기 발견하여
일정 지연 없이 완료했습니다."

---

**Q2. 요구사항 변경을 최소화하기 위한 방법은?**

답변 전략:
- 초기 요구사항 분석 철저히
- 프로토타입 또는 목업 제작
- 단계별 검수로 조기 피드백

모범 답변:
"요구사항 변경을 최소화하기 위해 3가지 방법을 적용합니다:

1. 초기 요구사항 정의 단계 강화 (1개월)
   - 발주처 담당자와 주 2회 미팅
   - 모든 요구사항을 문서화 및 확인

2. 프로토타입 제작 (개발 전)
   - 와이어프레임 또는 목업 제작
   - 사전에 화면 구성 확인 → 오해 방지

3. 단계별 검수 (2주 단위)
   - 완성된 기능 즉시 시연
   - 빠른 피드백으로 방향 수정

이렇게 하면 요구사항 변경을 50% 이상 줄일 수 있습니다."
```

```
당신은 까다로운 공무원 평가위원입니다.

- 회사: 중소기업, 설립 2년

- 실적: 공공 0건, 민간 2건
- 기술: Java, Vue.js (백엔드 경험 적음)

- "망하면 어쩔 건가?" (지속 가능성 의심)

[톤]
- 직설적이고 공격적
- 의심 가득한 어조
- 압박감 주는 질문

[출력 형식]
```

```markdown

오히려 첫 프로젝트이기에

대기업은 수십 개 프로젝트를 동시 진행하지만,
```
